 Product Search & Autocomplete (Trie + KMP): The search bar uses a Trie for real-time prefix autocomplete and KMP for full-text substring matching — chosen because Trie gives O(L+k) prefix lookups unmatched by any flat structure, and KMP avoids redundant character comparisons unlike naïve O(n·m) search; alternatives like Ternary Search Tree, Suffix Array, or Regex scanning were not chosen due to higher memory overhead, slower build times, or unpredictable worst-case performance.
Product Filtering & Price Range (BST + Binary Search): A BST indexes products by category/price for O(log n) range queries, paired with Binary Search for price-slider bounds — chosen because BST naturally maintains sorted order for range scans; alternatives like a sorted array with linear scan, skip list, or B-Tree were not chosen since linear scan is O(n), skip lists add pointer complexity, and B-Trees are overkill for in-browser use.
Product Sorting (Merge Sort): All product and client table sorting uses Merge Sort — chosen for its guaranteed O(n log n) stable sort that preserves equal-key order (e.g. same-price items keep their original sequence); alternatives Quick Sort and Heap Sort were included only as benchmark comparisons, not production sorters, because Quick Sort is unstable and has O(n²) worst case, while Heap Sort's cache-unfriendly access pattern makes it slower in practice.
Cart Management (HashMap + Stack): The cart uses a HashMap for O(1) add/remove/update by product ID, and a Stack for instant undo of the last cart action — chosen because no other structure matches O(1) keyed access, and a Stack's LIFO nature perfectly models undo history; alternatives like a linked list cart or array-based undo log were not chosen due to O(n) lookup and the overhead of scanning for undo targets.
Coupon Code Validation (HashMap): Coupon codes are stored and validated via HashMap in O(1) — chosen because discount lookup is a pure key→value problem with no ordering or range needs; alternatives like a sorted array + binary search or Trie of coupon strings were not chosen as they add unnecessary complexity for a fixed, small key set.
Order History (Doubly Linked List): Orders are stored in a Doubly Linked List for O(1) append and reverse traversal (newest-first display) — chosen because DLL gives O(1) head/tail insertion without reallocation; alternatives like a dynamic array or singly linked list were not chosen because arrays require O(n) shifts on prepend and singly linked lists can't traverse backwards efficiently.
Product Detail Caching (LRU Cache): Revisited product pages are served from an LRU Cache backed by a HashMap + DLL for O(1) get/put — chosen because LRU evicts the least-recently-used entry, matching real user browsing behavior; alternatives like FIFO cache, LFU cache, or no cache were not chosen since FIFO ignores recency, LFU is more complex to implement for marginal gain, and no cache means redundant re-renders.
Recommendations (Graph + BFS + Cosine Similarity): A product Graph weighted by tag-set similarity is traversed via BFS to collect recommendation candidates, ranked by Cosine Similarity — chosen because BFS guarantees shortest-hop discovery of related products and cosine similarity is a standard, computationally efficient collaborative-filtering metric; alternatives like DFS, matrix factorization, or pure tag matching were not chosen because DFS can go deep into irrelevant nodes, matrix factorization requires server-side ML, and plain tag matching ignores interaction-weighted relevance.
Reorder Alerts & Top-K Products (Min/Max Heap): A Min-Heap surfaces the lowest-stock items first for reorder urgency, while a Max-Heap extracts the top-K featured products — chosen because heaps give O(log n) insert and O(1) peek, making them ideal for priority queues; alternatives like sorting the full array each time (O(n log n) per update) or a sorted linked list were not chosen due to the cost of repeated full sorts and pointer overhead.
Delivery ETA Routing (Dijkstra's Algorithm): Shortest delivery path across Nairobi zones is computed with Dijkstra's Algorithm — chosen because it finds the provably optimal weighted shortest path in O((V+E) log V) on a non-negative-weight graph; alternatives like Bellman-Ford (slower at O(V·E), needed only for negative weights), A* (requires a good heuristic and more complex setup), or Floyd-Warshall (O(V³), impractical for real-time per-order use) were not chosen.
Revenue Analytics (Segment Tree + Sliding Window): A Segment Tree answers arbitrary revenue range queries in O(log n), while a Sliding Window Max computes rolling revenue trends in O(n) — chosen because the Segment Tree avoids re-aggregating data on every query, and the sliding window is optimal for fixed-window rolling stats; alternatives like prefix sum arrays (only support static data) or recomputing totals on every query (O(n) per query) were not chosen for performance reasons.
Client Search (KMP): Admin client search uses KMP for O(n+m) substring matching across names and emails — chosen because KMP's failure-function preprocessing eliminates backtracking, making it faster than naïve search on repeated patterns; alternatives like Rabin-Karp (hash collisions add complexity) or regex engine (unpredictable worst-case) were not chosen for reliability in a dependency-free app.
